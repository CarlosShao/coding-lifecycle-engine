# 对抗性审查（Adversarial Review）— 5 角色 Prompt

> **Fuel file** · 文章燃料（原话）· 注入阶段：**Phase 3 — 对抗验收**（★不糊弄★）
> 由 `code-review` 切 adversarial 模式调用。替代"帮我检查一下有没有问题"这种过宽、只能做表面检查的问法。
> 目的：让 AI 不再只站"开发者视角"看自己的代码，而从攻击者、异常用户、极端场景、系统故障角度重新审视产品。

---

## 1. 提示词（原文，照抄，不要改写措辞）

```
请开启对抗性审查。你现在分别扮演恶意用户、安全工程师、资深测试工程师、性能工程师和挑剔的真实用户，从不同角度寻找这个系统可能被攻击、绕过、误用或压垮的方式。请按严重程度列出问题，并给出复现路径、影响范围、修复建议和验证方式。
```

---

## 2. 五个角色各自 probing 什么

文章点名的五类攻击/失效面，分别对应五个角色：

| 角色 |  probing 角度 | 典型问题 |
|---|---|---|
| **恶意用户** | 权限与越权 | 权限是否可以被绕过？能否访问别人的数据 / 操作？ |
| **安全工程师** | 参数与注入 | 接口参数是否可以被篡改？是否有注入 / 越权写？ |
| **资深测试工程师** | 异常输入与崩溃 | 异常输入会不会导致系统崩溃？边界 / 空 / 超限怎么处理？ |
| **性能工程师** | 压垮与资源 | 高并发会不会出现数据错误、性能下降或资源耗尽？ |
| **挑剔的真实用户** | 误操作与中断 | 用户连续点击、重复提交、网络中断时，系统是否还能正常工作？ |

文章原文覆盖的具体场景：
- 权限是否可以被绕过？
- 接口参数是否可以被篡改？
- 异常输入会不会导致系统崩溃？
- 用户连续点击、重复提交、网络中断时，系统是否还能正常工作？
- 高并发时会不会出现数据错误、性能下降或资源耗尽？

## 3. 必输出格式（按严重程度排序的表格）

AI 必须输出一个**按严重程度从高到低排序**的问题表，每行包含四列固定字段：

```markdown
## 对抗性审查结果

| 严重程度 | 问题摘要 | 角色视角 | 复现路径 | 影响范围 | 修复建议 | 验证方式 |
|---|---|---|---|---|---|---|
| Critical | ... | 安全工程师 | 步骤1→2→3 | 所有用户数据 | ... | 补测试+手动验证 |
| High | ... | 性能工程师 | ... | ... | ... | ... |
| Medium | ... | 测试工程师 | ... | ... | ... | ... |
| Low | ... | 真实用户 | ... | ... | ... | ... |
```

字段定义（不可省略任一列）：
- **严重程度**：Critical / High / Medium / Low。
- **复现路径**：可一步步执行的操作，能稳定触发问题。
- **影响范围**：受影响的用户 / 数据 / 模块。
- **修复建议**：具体改法，不是"建议加强"。
- **验证方式**：如何证明修好了（测试用例 / 手动步骤 / 压测指标）。

---

## 4. 使用纪律

- 本审查是 Phase 3 的硬动作，跑在 `anti-shortcut-workflow` 逐条核查之前或之中。
- **不糊弄**：禁止把"基本没问题"当结论。每个 Critical/High 必须有复现路径与验证方式，否则视为审查未完成。
- 发现 bug → 走 `diagnosing-bugs` 的"假设→验证→修复→复验"闭环（见系统 plan §16.1），不得猜修。
- 弱模型若做不了 5 角色，降级为结构化 checklist（五角色拆成固定问题模板），但**不跳过验收**。
