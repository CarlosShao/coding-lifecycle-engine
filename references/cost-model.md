# Cost Model

> 机制文件 · token / 费用预算、超预算动作、模型路由、弱模型降级。成本目标是**可见、可告警、可追责**，不是硬阻断。

## 1. 预算阈值（默认，可配置）

写入 `projects` 配置或 `set_convention`，按 `maturity` 给合理默认值：

| 档位 | `maturity` | 默认预算 (USD) | 说明 |
|---|---|---|---|
| MVP | `mvp` | **$50** | 单人或小验证，控制烧钱 |
| Commercial | `commercial` | **$200** | 可交付产品，允许更大探索 |
| Enterprise | `enterprise` | **$1000** | 长期项目，多会话并行 |

预算是**项目全生命周期**累计（来自 `cost_ledger` 的 `cost_usd` 求和），不是单任务。可按 `task_id` 下钻查哪步最贵。

## 2. 超预算动作：告警，不阻断

- `record_cost` 写入后，server 累计并比对阈值。
- 触发点（满足任一即**告警**）：
  - 达到 **80%** → `WARN`：提示"已用 X/Y，后续多用便宜模型"。
  - 达到 **100%** → `OVER_BUDGET`：提示用户，但**不阻止继续**——由人类决定加预算或收尾。
  - 超过 **150%** → `OVER_BUDGET_HARD`：再次强提示，仍不自动停（系统无权限 kill 会话）。
- 告警写入 `events` 表，并在下次 `get_state` 显式回显。超预算**绝不**静默吞掉——违反"诚实边界"纪律。

> 理由：硬阻断会卡死验收（如对抗验收正好最贵）。预算是治理信号，不是开关。

## 3. 模型路由矩阵（route_model）

`route_model(phase, task_type)` 返回推荐模型。原则：**推理重的少做、量大的便宜做**。

| 阶段 / 动作 | task_type | 推荐模型档 | 理由 |
|---|---|---|---|
| Phase 0.5 | `grilling` | **强**（高推理） | 压力测试方案，需深度推理 |
| Phase 3 | `adversarial` | **强** | 多角色对抗审查，需上下文广 |
| Phase 4 | `pre_mortem` | **强** | 失败预演需因果推理 |
| Phase 1 | `decompose` / `freeze_contract` | 中 | 结构拆分，中等推理 |
| Phase 2 | `tdd` / `implement` | **便宜** | 量大、模式化，便宜模型足够 |
| Phase 2 | `code_review` (非对抗) | 中 | 常规 review |
| 任意 | `bug` 诊断循环 | 中→强 | 视假设复杂度升档 |
| 任意 | `refactor` / `docs` | 便宜 | 低风险、模式化 |

具体模型名在实施时定（见 §16.13：属主观调参项，用合理默认 + 可配置）。路由结果仅**建议**，agent 可据实际可用性降级。

## 4. 弱模型降级（永不跳过验证）

当强模型不可用（配额/不可用/本地弱模型），系统**降级为结构化 checklist 兜底**，不跳过验收：

- 对抗验收降级：五角色（恶意用户/安全/测试/性能/挑剔用户）拆成**固定问题模板**，逐题填答 + 工具证据，替代自由多角色推理。模板见 `adversarial-review.md` 的退化模式。
- grilling 降级：用 `first-principles.md` 的固化检查单（本质/约束/风险/边界）逐项追问，不靠自由对话压力测试。
- pre-mortem 降级：用 `pre-mortem.md` 的分类清单（用户需求/技术债/性能/数据增长/依赖/协作/商业）逐类打分，不靠自由推演。

**铁律**：降级只降"推理自由度"，**不降"验证严格度"**。任何验收仍须有工具证据（lint 通过、测试绿、覆盖达标）。弱模型写不出证据 → 该 checklist 项判 `fail`，不准 `pass`。
