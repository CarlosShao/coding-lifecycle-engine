# Parallel Integration

> 机制文件 · 并行开发的机械细节：模块所有权、共享文件锁、冲突解决、强制集成步、契约变更升级。

## 1. 模块所有权不重叠（硬前提）

- Phase 2 启动前，每个 `atomic_task` 必须归属单一 `module`（auth/api/ui/db/...），且认领它的 session 拥有该模块**独占所有权**。
- 不同 session 认领不同模块 → 天然不冲突。
- `claim_task` 由 server 加锁：已被认领 → 拒绝，返回"已被 Session X 认领"。
- 同一模块内多任务由**同一 session 顺序**完成，不跨会话拆同一模块（避免内部冲突）。

## 2. 共享文件锁（模块外仍有交集文件）

模块不重叠仍会碰共享文件：路由注册、共享类型、入口 `main`、全局配置。这些**不归任何单一模块独占**，必须走文件锁串行化。

| 工具 | 调用时机 | 行为 |
|---|---|---|
| `acquire_file_lock(file_path, session)` | 准备写共享文件前 | 获锁；被占 → 返回 owner，调用方**等待或改本地副本** |
| `check_file_lock(file_path)` | 写前确认 | 返回当前 owner 或空闲 |
| `release_file_lock(file_path, session)` | 写完立即 | 释放；owner 不符 → 拒绝 |

**规则**
1. 写任何"跨模块共享文件"前**必须** `acquire_file_lock`，写完**立即** `release_file_lock`。
2. 锁只保护写，读不锁。
3. 锁是短期互斥，不是长事务；持有期间不调外部阻塞调用。
4. 冲突一方被拒时**不得**自行改文件名绕过（如 `main_copy.ts`）——那制造分叉。等待或报告冲突。

## 3. 冲突解决协议

当集成时发现两模块对同一接口/数据理解不一致：

1. **识别**：`integrate_changes` 跑冲突检测（同名导出、契约字段不符、类型不匹配）。
2. **裁决依据**：以 `contracts` 表中 `status='frozen'` 的接口为准——它是单一真相源。
3. **修复**：偏离契约的一方回改对齐契约；**谁偏离谁改**，不是各退一步。
4. **不可私下改契约**（见 §5）：若发现契约本身有错，走升级，不在集成时静默改。

## 4. 强制集成步（integrate_changes）

并行开发后**必须**有集成负责人（`integrate_changes` 指定，人或 agent），执行：

- [ ] 合并所有模块分支/变更到集成基线
- [ ] 跑冲突检测，按 §3 解决
- [ ] **冒烟测试清单**（全绿才允许进 Phase 3）：
  - [ ] 应用能启动（无启动期 panic / 导入错误）
  - [ ] 健康端点 / 入口路由可达
  - [ ] 跨模块主链路冒烟（如 login → 业务动作 → 返回）走通
  - [ ] 共享类型编译/类型检查通过
  - [ ] 契约一致性校验：实际实现 vs `contracts` 冻结 spec 无偏差
- 集成结果写入 `events` + `progress_log`；冒烟未全绿 → 标记 `blocked`，不进 Phase 3。

## 5. 冻结契约变更升级（禁静默改）

并行中发现冻结契约需改 → **一律禁止就地编辑**，必须升级：

- 调用 `escalate_to='phase_1'`（契约层错）触发：
  - 回 Phase 0.5 grilling 重审（若属方案层错），或
  - Phase 1 重 `decompose` + `contracts` 版本 +1（接口变更）。
- 受影响 ticket 自动 `blocked`（依赖新契约版本）。
- 升级记录入 `approvals`（`architecture_approval` 门），新契约重新 `freeze_contract` 后才解禁。

> 理由：契约是并行安全的根基。任何人静默改契约，其他模块基于旧契约的实现全部失效且无声——这是并行开发最隐蔽的灾难。
