# 计划文档纪律（External Memory Discipline）— 外部记忆

> **Fuel file** · 文章燃料（原话精神）· 贯穿 **Phase 1 → Phase 2 并行开发**
> 这是文章里"几乎必做"的动作：每完成一小项任务，就更新计划文档。本文件规定更新什么、为什么、何时更新。

---

## 1. 核心动作（原文精神）

文章原话：

> 每完成一小项任务，都更新计划文档中的完成状态、已做决策、遗留问题和下一步行动。这件事看起来简单，但真的能避免很多返工。

四个必填字段（每次更新都写全）：

- **完成状态**：刚做完的那一小项现在是什么状态（done / 部分 done）。
- **已做决策**：这次顺手定了什么（选型、命名、边界取舍）——不记就会忘。
- **遗留问题**：卡住的点、未决项、发现的新风险。
- **下一步行动**：接下来该做哪一小项，谁来做。

---

## 2. 为什么必须做（三理由，原文依据）

1. **AI 会做到一半忘记做到哪**
   > 因为项目一旦复杂，AI 很容易出现"做着做着忘记做到哪里"的情况。
2. **会话可能中断 / 上下文过长**
   > 再加上会话可能中断、上下文过长……
3. **可能要另开新会话**
   > ……或者你需要另开一个新会话，如果没有一份持续更新的计划文档，新的 AI 就要重新理解整个项目。

文章总结：这份文档相当于给项目留下**"外部记忆"**。哪怕会话中断，也可以直接告诉新的 AI：

> 请先阅读计划文档，确认当前完成状态、遗留问题和下一步任务，然后继续执行。

这样 AI 就能快速接上进度，而不是从头开始。

---

## 3. 计划文档模板（每个工作项 / 任务一块）

```markdown
# 计划文档：<项目名称>

## 任务 #<id> — <标题>
- **完成状态**：<done | in_progress | blocked>
- **已做决策**：
  - <决策1：为什么这么定>
  - <决策2>
- **遗留问题**：
  - <未决项 / 风险 / 卡点>
- **下一步行动**：
  - <下一小项 + 负责视角>

## 进度叙事（progress_log，每完成一小项追加一行）
- [时间戳] <做了什么，验证结果>
```

> 对应系统 plan §2.5 的 `tasks.progress_log` 字段——叙事进度每完成一小项追加，落实本纪律。

---

## 4. 更新规则（何时必须写）

| 触发时机 | 动作 |
|---|---|
| **每完成一个原子任务后** | 立即更新该任务的 完成状态 / 已做决策 / 遗留问题 / 下一步行动，追加 progress_log 一行。不允许"攒一批再写"。 |
| **做出任何决策时** | 当场写入"已做决策"，哪怕很小（命名、边界取舍）。决策不进文档 = 等于没做。 |
| **发现遗留 / 新风险时** | 立即写入"遗留问题"，必要时 `block_task` 记 blocked 并写 workaround。 |
| **结束一个会话前** | 确保最后状态已落盘（plan doc + lifecycle.db）。新会话靠读文档续跑，不靠记忆。 |
| **会话中断 / 上下文将满前** | 强制写 STATE + 生成 handoff 笔记（见系统 plan §16.8 会话自动 checkpoint）。 |

---

## 5. 纪律红线

- **不假装通过**：外部依赖阻塞时，写 `blocked` + 真实原因，绝不写"已集成通过"（系统 plan §0.3 纪律②）。
- **不重启理解**：新会话第一件事是读计划文档，不是重新问需求。
- **不拖延**：小任务完成到文档更新之间不应有长间隔；AI 动作"先写日志再动手"优于"做完再补"。
