# PRD Prompt — 产品经理视角

> **Fuel file** · 文章燃料（原话）· 注入阶段：**Phase 0 — 需求与初始方案**
> 由 `to-spec` 引擎调用，产出 PRD 后进入技术经理视角（见 `tech-plan-prompt.md`）。
> 本文件第一阶段运行于 **grilling 打磨之前**——先把模糊想法梳理成完整文档，再让方案被压力测试。

---

## 1. 提示词（原文，照抄，不要改写措辞）

```
你是一名资深产品经理，请基于以下需求，输出一份完整的产品需求文档。内容包括：项目背景、目标用户、核心问题、功能范围、用户流程、页面与交互说明、边界条件、非功能性要求、验收标准，以及暂不纳入本期的内容。

需求：***
```

> 把 `***` 替换成用户的一句话想法或原始需求片段。若用户给的是已有代码库（existing 项目），先用 `codebase-design` 读现有代码，再让 AI 站在"在现有系统上增加/改造什么"的角度输出 PRD。

---

## 2. PRD 输出模板（AI 应严格按此结构填充）

产出必须是可直接评审的 Markdown，章节顺序固定，不允许漏项：

```markdown
# 产品需求文档（PRD）：<项目名称>

## 1. 项目背景
- 为什么要做：业务/用户痛点来源
- 当前现状与缺口
- 预期收益（定性即可）

## 2. 目标用户
- 主要用户画像（角色、场景）
- 次要用户 / 利益相关方

## 3. 核心问题
- 用户真正想解决的一件事（一句话）
- 不解决会怎样

## 4. 功能范围
- 本期必须做（MUST）
- 本期可不做但相关（SHOULD / MAY）
- 用清单，避免散文

## 5. 用户流程
- 主流程（happy path）分步
- 关键分支流程
- 可用文字步骤，不必画图

## 6. 页面与交互说明
- 核心页面清单
- 每页关键元素与交互行为
- 状态（加载/空/错误/成功）如何处理

## 7. 边界条件
- 异常输入、空数据、超限、并发、弱网
- 权限边界（谁能看/能改什么）

## 8. 非功能性要求（NFR）
- 性能 / 延迟 / 吞吐预期
- 兼容性 / 可访问性
- 安全与合规底线

## 9. 验收标准
- 每条可测、可勾选（Given/When/Then 风格）
- 与"功能范围 MUST"一一对应

## 10. 暂不纳入本期的内容
- 明确划出范围外，避免后期膨胀
```

---

## 3. 使用纪律

- **顺序不可颠倒**：先出 PRD，再出技术方案。不要一上来就写代码。
- **grilling 前产物**：本 PRD 是 Phase 0.5 grilling 的被审对象。grilling 通过（`pass_gate`）前，技术方案不得冻结。
- **模糊需求处理**：若用户只给"我想做一个 XX"，让 AI 用产品经理视角补全背景与用户，而不是追问一堆问题卡住流程。
- **原子性**：PRD 写清后，Phase 1 才能据此 `decompose` 拆任务。PRD 含糊 → 拆出来的任务必然含糊。
