# Tech Plan Prompt — 技术经理视角（含第一性原理）

> **Fuel file** · 文章燃料（原话）· 注入阶段：**Phase 0 — 需求与初始方案**（PRD 之后）
> 由 `codebase-design` 引擎调用，产出可执行技术方案 + 栈选型草案。
> **关键约束**：`从第一性原理出发` 这一指令**必须始终出现在提示词中**，不可省略、不可弱化。

---

## 1. 提示词（原文，照抄，不要改写措辞）

```
你是一名资深技术经理，请基于这份产品需求文档，输出一份可执行的技术方案。请从第一性原理出发，拆解问题本质、核心约束和关键风险，再给出系统架构、技术选型、数据模型、接口设计、开发拆分、依赖关系、测试方案和上线计划。
```

> 输入是上一步 `prd-prompt.md` 的产出。若涉及现有代码库，先读现有架构再写；栈选型参考 `stack-decision.md`。

---

## 2. 为什么"从第一性原理出发"不可删

这篇文章明确指出：第一性原理**不是让 AI 堆砌专业术语**，而是强制它回到问题本身。删掉这句，AI 会直接落到"习惯性技术栈 + 模板化架构"，跳过对问题本质的思考，后期才发现边界被忽略。

- 必须保留原话中的推导顺序：**先拆解问题本质 → 核心约束 → 关键风险**，然后才给架构与选型。
- 若 AI 一上来就列技术栈，视为偏离指令，应要求它回到"问题本质"段落补写。

详见 `first-principles.md`（第一性原理定义与 checklist）。

---

## 3. 技术方案输出模板（AI 应严格按此结构填充）

```markdown
# 技术方案：<项目名称>

## 1. 问题本质（第一性原理）
- 用户真正要解决的是什么（剥离手段，只看目标）
- 最核心的输入 / 输出 / 约束
- 哪些设计是必要的，哪些只是习惯性方案（明确标出"可质疑项"）

## 2. 核心约束
- 硬约束（性能、合规、既有系统、预算、时间）
- 软约束（团队熟悉度、可维护性）

## 3. 关键风险
- 风险清单 + 发生概率 + 影响 + 初步缓解

## 4. 系统架构
- 模块划分与职责边界
- 关键数据流（文字或简图）

## 5. 技术选型
- 语言 / 框架 / 存储 / 基础设施 + 选择理由（对照第一性原理）
- 明确"为什么不用 XX"

## 6. 数据模型
- 核心实体与关系
- 关键字段与约束

## 7. 接口设计
- 模块间接口（冻结前先定义）
- 对外 API 契约要点

## 8. 开发拆分
- 拆成可并行的小任务（对应 Phase 1 DAG）
- 每任务负责人视角（前端/后端/测试/审查）

## 9. 依赖关系
- 任务间 depends_on / blocks
- 外部依赖与第三方

## 10. 测试方案
- 单元 / 集成 / 端到端覆盖策略
- 边界与异常用例

## 11. 上线计划
- 部署路径（参考 deploy-checklist / ops-checklist）
- 灰度 / 回滚预案
```

---

## 4. 使用纪律

- **第一性原理指令是硬门槛**：提示词模板固定，禁止任何"精简版"删掉该句。
- **先于并行**：技术方案 + 冻结契约齐全后，才进入 Phase 2 并行开发。
- **被 grilling 审**：本方案是 Phase 0.5 grilling 的评审对象，锁定前不会被采纳。
