# Test Fixtures Guide

> 机制文件 · 测试数据 / factory / fixture 管理；种子数据；测试环境与生产环境一致性；测试间隔离；禁止共享可变状态。

## 1. 三条不可妥协的纪律

1. **隔离**：每个测试拥有自己的数据世界，互不读写对方状态。测试顺序无关、可单独跑。
2. **一致**：测试环境的数据形状/约束/配置必须与生产同源（同一 schema、同一校验规则），否则测试过了生产崩。
3. **无共享可变状态**：全局单例、模块级可变变量、静态缓存——测试禁止共享，必须每测试重建或注入假对象。

## 2. Fixture / Factory 模式

- **Factory（首选）**：用工厂函数动态造对象（如 `makeUser(overrides)`），而非手写大段 JSON fixture。
  - 工厂产出**最小合法对象** + 允许覆盖字段，避免" fixture 改一处全测试崩"。
  - 工厂放 `tests/factories/` 或等效，按领域分组。
- **Fixture（静态）**：仅用于固定参照数据（如标准税率表、字典），放 `tests/fixtures/`，**只读**。
- 禁止：测试直接连生产库造数据、手写裸 SQL 混在测试逻辑里、复制粘贴大段对象字面量。

## 3. 种子数据（seed）

- 种子数据用于：本地起项目即有可演示数据、集成冒烟有基准。来源单一：`db-migration-guide.md` 的种子脚本或 `seed/*.sql|*.ts`。
- 种子数据**必须**走与生产相同的校验/约束（同 schema、同触发器）。
- 测试用种子应是**子集或独立命名空间**，不与生产种子混库。测试库 ≠ 生产库。
- 种子变更要随 schema 迁移一起评审，不偷偷改。

## 4. 测试环境 vs 生产环境一致性

| 维度 | 要求 |
|---|---|
| Schema | 测试库 schema == 生产 schema（同源迁移，禁止测试库手改字段） |
| 校验规则 | 同一套（非空/唯一/外键/枚举），不在测试放宽 |
| 配置 | 用同一 config 加载路径，仅值不同（如 `TEST_DB_URL`）；不准测试里硬编码绕过校验 |
| 依赖行为 | mock 外部依赖时，**契约**须对齐真实（响应结构/错误码），mock 通过 ≠ 集成通过（§0.3 纪律2） |
| 时区/locale | 固定（如 UTC），避免环境差异导致 flaky |

## 5. 测试间隔离的具体做法

- **每测试 setup/teardown**：setup 建独立数据集，teardown 清掉（事务回滚最优：`BEGIN` → 测 → `ROLLBACK`，不留痕）。
- **唯一资源**：需要端口/文件/队列时，用随机端口或 `tmpdir` + 唯一名，不抢固定资源（防 flaky，见 `flaky-test-protocol.md`）。
- **无全局可变**：模块级 `let cache = {}` 类状态，测试前 `reset()` 或依赖注入替换。
- **并行安全**：测试标注可并行时，确保不共享 DB 行/文件/环境变量。

## 6. 与 flaky / bug 的关联

- 多数 flaky 根因是**违反本节隔离**（共享状态/顺序依赖），排查见 `flaky-test-protocol.md` Step 3。
- 测试 fixture 造出的"假数据"若偏离真实分布，会漏真 bug——工厂 override 要覆盖边界值，不只 happy path。
